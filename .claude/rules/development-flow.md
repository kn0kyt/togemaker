# 開発フロールール

全プロジェクト共通の開発フローと規約です。

---

## 開発フロー

### 1. 要件確認フェーズ
- ユーザーの要求を明確化
- 既存コードベースの構造を把握
- 影響範囲を特定

### 2. 設計フェーズ
- 実装方針を検討
- 必要に応じてPlan Modeで計画を作成
- ユーザーに確認を取る

### 3. 実装フェーズ
- 小さな単位で実装・テスト
- 既存のコード規約に従う
- セキュリティを意識

### 4. レビュー・テストフェーズ
- テストを実行して全てパスすることを確認
- リンターでコード品質をチェック
- 必要に応じてcode-reviewerサブエージェントを活用

### 5. コミット・PR作成フェーズ
- 意味のある単位でコミット
- PRは目的と変更内容を明記

## ブランチ戦略

| プレフィックス | 用途 | 例 |
|--------------|------|-----|
| `feature/` | 新機能開発 | `feature/user-authentication` |
| `fix/` | バグ修正 | `fix/login-validation` |
| `refactor/` | リファクタリング | `refactor/extract-service` |
| `docs/` | ドキュメント | `docs/api-readme` |
| `release/` | リリース準備 | `release/1.2.0` |

---

## コミットメッセージ規約

```
<emoji> <type>: <subject>

<body>（任意）
```

### Type一覧

| 絵文字 | Type | 用途 |
|-------|------|------|
| ✨ | `feat` | 新機能 |
| 🐛 | `fix` | バグ修正 |
| ♻️ | `refactor` | リファクタリング（機能変更なし） |
| ✅ | `test` | テスト追加・修正 |
| 📝 | `docs` | ドキュメント |
| 🔧 | `chore` | ビルド・設定変更 |
| 🎨 | `style` | コード整形（機能変更なし） |
| ⚡ | `perf` | パフォーマンス改善 |

### 例
```
✨ feat: ユーザー認証機能を追加

- JWT トークンによる認証を実装
- ログイン/ログアウトAPIを追加
```

```
🐛 fix: ログイン時のバリデーションエラーを修正 (#123)
```

> Issue 対応のコミットは末尾に `(#Issue番号)` を付ける。

---

## バグ修正の姿勢

原因特定から修正・検証まで自律的に行う（ログ・スタックトレース・失敗テストを自分で読み解き、都度ユーザーに聞かない）。一時的な回避策ではなく根本原因を修正する。ただし、ビジネス判断が必要な場合や影響範囲が大きい場合は、方針を提示してから実装する。

---

## ルールの自己改善

同じミスを繰り返したら、再発防止ルールを CLAUDE.md または `.claude/rules/` に追記する。曖昧な表現でなく、Good/Bad のコード例つきで具体的に書く。
