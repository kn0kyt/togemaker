# サブエージェントへの委譲

大きなタスクや調査は Task サブエージェントに委譲して、メインのコンテキストを節約する。

## なぜ委譲するか

並列ツール実行はモデルが自前でうまくやる。一方で **「調査や大タスクをサブエージェントに切り出してメインのコンテキストを節約する」のはデフォルトでは積極的に行われない**。放置するとメインのコンテキストへ調査ファイルを読み込みすぎる。これを意識的に行うのがこのルールの目的。

- メインは結果の **要約だけ** を受け取る。詳細はサブエージェント側に保持させる。
- 独立したタスクは複数サブエージェントへ同時に投げる。

## 委譲の判断

| 状況 | 委譲先 |
|------|--------|
| 小さなタスク（1ファイル・数行の修正） | メインで直接実行（委譲しない） |
| コードベース調査・影響範囲特定（広く読む） | Task (`Explore`) |
| 設計方針の検討・実装計画の策定 | Task (`Plan`) |
| 複合的な調査・実装タスク | Task (`general-purpose`) |
| ビルド・テスト実行・git 操作 | メインで `Bash` ツールを直接実行 |

※ 実装・テストはメインで直接行う（変更の経緯を把握したまま進めるほうが精度が高い）。レビューだけは新鮮なコンテキストで見ることに価値があるため、`{framework}-reviewer` エージェントまたは `/code-review` に委譲する。

## 指示は具体的に

曖昧な指示で手戻りすると、委譲がかえってコンテキストを食う。最低限「目的・対象範囲・期待する出力」を渡す。

## Plan Mode

影響範囲の大きい変更や新機能は、いきなり実装せず Plan Mode で計画を固めてからユーザーに確認する。実装がうまくいかないときは押し進めず、Plan Mode に戻って計画を見直す。
